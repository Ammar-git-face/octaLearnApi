## Abstract

Naijabus is a full-stack platform designed to modernize intercity bus travel in Nigeria. It combines passenger-facing booking (web & mobile), driver/agent apps, and an operator admin dashboard with real-time tracking, dynamic seat allocation, digital payments, and analytics. The platform targets pain points such as inefficient ticketing, long queues, lack of transparency in seat availability and pricing, poor fleet utilization, and unsafe travel information.

This technical research document outlines the problem, proposed solution, system architecture, data model, API surface, security and privacy considerations, deployment and scaling strategies, testing approaches, demo plan for a hackathon, and recommended next steps.

---

## 1. Problem Statement

Intercity bus travel in Nigeria faces several systemic challenges:

- Manual/fragmented ticketing processes with paper tickets or cash-only payments.
- Unclear seat availability leading to overbooking or underutilization.
- Long queues and poor customer experience at terminals.
- Lack of real-time vehicle tracking and ETA information for passengers.
- Difficulty for operators to manage schedules, drivers, and maintenance.
- Limited data insights about routes, occupancy, and revenues.

Naijabus aims to address these problems by creating an integrated digital platform that benefits passengers, operators, and drivers.

## 2. Target Users & Stakeholders

- **Passengers:** Book seats, receive e-tickets, track buses, and pay online.
- **Bus Operators / Fleet Managers:** Manage routes, trips, vehicles, drivers; view analytics and revenue.
- **Terminal Agents / Box Office:** Sell/validate tickets, view passenger lists.
- **Drivers / Conductors:** Receive assignments, navigate to pickup points, report trip status via driver app.
- **Payment Processors / Banks:** Handle secure card/mobile payments.
- **Regulators / Transport Authorities (optional):** Access anonymized data for planning.

## 3. Product Goals & Key Features (MVP vs. Future)

### MVP (Hackathon-friendly, core value):
- Passenger web app (responsive) + simple mobile PWA.
- Search routes, select seats from a visual seat-map, book and pay (card/USSD/QR).
- Generate e-ticket (QR code) and email/SMS confirmation.
- Operator dashboard: create routes, schedule trips, view bookings.
- Driver app (minimal): receive assigned trip details and update trip status.
- Basic real-time location tracking using GPS pings (mockable in hackathon).
- Admin authentication and role-based access control.

### Future / Advanced:
- Dynamic pricing (surge & discounts), loyalty programs.
- Route optimization and demand forecasting using ML.
- Multi-operator marketplace & aggregator model.
- Offline mode for drivers/agents, cash reconciliation tools.
- Integration with national ID systems / KYC for higher trust.
- Full PCI-DSS compliance and audits.

## 4. High-level Architecture

**Layers:**
- Client: Passenger Web (Next.js or React PWA), Driver App (React Native or Ionic), Operator Dashboard (React + shadcn).
- API: Backend REST/GraphQL microservices (Node.js/Express or Fastify; or NestJS for structure).
- Data: Relational DB (PostgreSQL) for transactional data; Redis for caching & rate-limiting; Timeseries DB or event store for telemetry (optional).
- Real-time: WebSocket or Pub/Sub (Socket.io, Pusher, or Ably) for live updates and driver location.
- Payments: Payment gateway integrations (Flutterwave, Paystack, Stripe where available).
- Infra: Containerized services (Docker) orchestrated by Kubernetes, CI/CD pipelines, monitoring and logging.

**Component Diagram (textual):**
- Client Apps <-> API Gateway (JWT auth, rate-limit) -> Backend Services (Auth, Booking, Payments, Fleet, Notifications)
- Backend -> PostgreSQL (master/replica)
- Backend -> Redis (session, cache, seat-lock)
- Backend -> Message Broker (RabbitMQ / Kafka) for background jobs (email, reconciliation)
- Telemetry: Prometheus + Grafana

## 5. Data Model (Core Tables)

Below are suggested core tables (Postgres schema simplified):

- **users**: id, name, email, phone, password_hash, role (passenger, operator, driver, agent), kyc_status, created_at
- **operators**: id, name, contact_info, country, city, rating
- **vehicles**: id, operator_id, plate_no, model, capacity, features (AC/WiFi), status
- **drivers**: id, user_id, operator_id, licence_no, phone
- **routes**: id, origin, destination, distance_km, expected_duration_min
- **trips**: id, route_id, vehicle_id, operator_id, departure_time, arrival_time, status (scheduled, boarding, enroute, completed, cancelled)
- **seats**: id, vehicle_id, seat_no, seat_type
- **bookings**: id, trip_id, user_id, seat_ids (array), status (booked, paid, cancelled, checked_in), price_total
- **payments**: id, booking_id, amount, currency, provider, provider_reference, status, paid_at
- **tickets**: id, booking_id, ticket_code, qr_code_data, issued_at
- **locations**: id, trip_id, lat, lng, recorded_at (for telemetry)
- **feedbacks**: id, user_id, trip_id, rating, comment

**Important design note:** Use seat-locking (Redis) to avoid race conditions during concurrent seat selection.

## 6. API Design (Representative Endpoints)

**Auth & Users**
- `POST /api/auth/register` — register user
- `POST /api/auth/login` — login (returns JWT)
- `GET /api/users/me` — get current user

**Search & Trips**
- `GET /api/routes?origin=...&destination=...&date=...` — search available trips
- `GET /api/trips/:id` — trip detail (seat map + availability)

**Bookings**
- `POST /api/bookings/hold` — create a provisional hold on selected seats (uses Redis)
- `POST /api/bookings/confirm` — confirm booking + initiate payment
- `POST /api/bookings/:id/cancel` — cancel
- `GET /api/bookings/:id/ticket` — retrieve e-ticket (QR)

**Payments**
- `POST /api/payments/webhook` — payment provider webhook
- `GET /api/payments/:id/status` — check payment status

**Operator / Admin**
- `POST /api/operators/routes` — create route
- `POST /api/operators/trips` — schedule trip
- `GET /api/operators/trips/:id/bookings` — view bookings
- `POST /api/operators/vehicles` — register vehicle

**Driver App**
- `GET /api/driver/trips` — assigned trips
- `POST /api/driver/trips/:id/location` — update GPS ping
- `POST /api/driver/trips/:id/status` — update trip status

**Notifications**
- `POST /api/notify/sms` — send SMS (via Twilio/MSG91)
- `POST /api/notify/email` — send email (SendGrid/Mailgun)

## 8. Concurrency & Consistency (Seat Booking)

Key challenge: concurrent seat selection can cause double-booking.

**Approach:**
- Use Redis-based seat lock with TTL (e.g., lock for 5–10 minutes while payment completes).
- On `hold` API, lock seat IDs for a user session; on `confirm`, persist to DB and remove lock; on TTL expiry, release seats.
- Use DB transactions for final confirmation to ensure atomicity.

Also apply optimistic concurrency control with version numbers for trips or seat maps if needed.

---

## 9. Payments & Reconciliation

- Support multiple payment methods: card (Paystack/Flutterwave/Stripe), bank transfer, USSD, POS at terminal.
- Implement webhook handlers for payment confirmation from providers.
- Create daily reconciliation jobs to verify payments vs. bookings, using message queues for eventual consistency.
- Store provider transaction references and full webhook payloads for audit.

Security: do not store full card data; use hosted checkout or tokenization. Aim for PCI scope minimization.

---

## 10. Security & Privacy

- Use HTTPS for all services; HSTS headers.
- JWT for stateless auth; refresh tokens stored securely using httpOnly cookies for web or secure storage on mobile.
- Rate-limiting and IP-based throttling on login and booking endpoints.
- Input validation and prepared statements to prevent SQL injection.
- Content Security Policy (CSP) for web clients.
- Encrypt sensitive data at rest (e.g., PII) and in transit.
- Secure payment integration; minimize PCI exposure (use payment providers' hosted fields).
- Role-based access control for operator/admin endpoints.
- Audit logs for critical operations (cancellations, refunds).

Privacy: conform with local data protection norms; provide privacy policy and data deletion processes.

- End-to-end tests for UI flows using Playwright or Cypress.

## 13. Deployment & DevOps

- Containerize services with Docker.
- Use managed Postgres (e.g., Supabase, AWS RDS) for reliability.
- CI/CD pipeline: Run tests and linting on PRs; auto-deploy to staging; manual promotion to production.
- Use Infrastructure as Code (Terraform) for reproducible infra.
- Use auto-scaling groups or k8s Horizontal Pod Autoscaler to handle spikes.

---

## 14. Scalability & Cost Considerations

- Start with a monolith API if team size and timeline are small; split into microservices as load increases.
- Cache hot route/trip data in Redis to reduce DB load.
- Use background workers for non-critical tasks (email, reporting).
- Consider CDN for static assets (web app) and WAF for basic security.
- Cost tradeoffs: managed services reduce ops effort but increase monthly costs; compute and DB are the primary recurring costs.

## 17. Risks & Mitigations

- **Risk:** Double booking & race conditions. **Mitigation:** Redis seat-lock, DB transactions, TTLs.

- **Risk:** Driver connectivity issues. **Mitigation:** Offline caching in driver app, store-and-forward telemetry.
- **Risk:** Regulatory / license issues. **Mitigation:** Engage local operators early and comply with transport rules.

## 21. Appendix — Example Sequence: Booking Flow

1. Passenger searches for route & date.
2. System returns available trips and seat maps.
3. Passenger selects seats and calls `POST /bookings/hold` — system locks selected seats in Redis.
4. Passenger proceeds to payment; frontend uses hosted checkout from payment provider.
5. Payment provider calls webhook to `POST /payments/webhook` on success.
6. Backend confirms booking in DB, generates ticket with QR and sends email/SMS.
7. Operator dashboard updates and driver sees passenger manifest in their app.
